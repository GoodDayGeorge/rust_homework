<h3><a id="user-content-modules" class="anchor" aria-hidden="true" href="#modules"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Modules</h3>
<p>Before you go any further, a word on modules and crates.</p>
<p>Crates are any Rust library or package. Modules are the inner logical sections
of a crate.</p>
<p>You will want to organize each section of your crate into a different module,
so consumers can only import the parts that they need. You could do this in
<code>lib.rs</code>:</p>
<div class="highlight highlight-source-rust"><pre><span class="pl-k">pub</span> <span class="pl-k">mod</span> problem1 {
}

<span class="pl-k">pub</span> <span class="pl-k">mod</span> problem2 {
}

<span class="pl-k">pub</span> <span class="pl-k">mod</span> problem3 {
}

<span class="pl-k">pub</span> <span class="pl-k">mod</span> problem4 {
}

<span class="pl-c">// ...</span></pre></div>
<p>But this gets unwieldy pretty fast. You can instead put these modules in
separate files:</p>
<pre><code>hw01
├── Cargo.toml
└── src
    ├── problem1.rs
    ├── problem2.rs
    ├── problem3.rs
    ├── problem4.rs
    └── lib.rs
</code></pre>
<p>Every <code>.rs</code> file defines a module that is the same as its filename, so
<code>problem1.rs</code> implicitly defines the module <code>problem1</code>.</p>
<p>Crates are organized into trees of files, where one file is the root, typically
<code>src/lib.rs</code> (or <code>src/main.rs</code>). To include a module, you need to declare it in
the crate root. To add <code>problem1</code> as a module, add the line <code>pub mod problem1;</code>
to the top of <code>src/lib.rs</code>. (This is equivalent to defining a module in the
file itself, with <code>pub mod problem1 { ... }</code>.) Until you add this directive,
Cargo will not try to build <code>problem1</code> part of your crate. The <code>pub</code> keyword in
this directive exposes the module <code>problem1</code> to any other crate that imports
your library. You can omit <code>pub</code> to leave modules private; however, if a
function is not exported or used internally, it will emit a dead code warning.</p>
<p>Within a module, all members (functions, types, submodules) are private by
default. The <code>pub</code> keyword can also be used to make any of these available from
outside of the module.</p>
<div class="highlight highlight-source-rust"><pre><span class="pl-c">// problem1.rs</span>

<span class="pl-c">/// Functions are private (only available to this module) by default.</span>
<span class="pl-c">/// Use the `pub` keyword to mark this function as public.</span>
<span class="pl-k">pub</span> <span class="pl-k">fn</span> <span class="pl-en">sum</span>(slice: <span class="pl-k">&amp;</span>[<span class="pl-k">i32</span>]) -&gt; <span class="pl-k">i32</span> {
    <span class="pl-c">// ...</span>
}</pre></div>
<p>There are a few different ways to import items from a different module:</p>
<ol>
<li>
<p>To add <code>sum</code> to the scope of a file, add the line <code>use problem1::sum</code> to the
top of the file. You can call the function with <code>sum()</code>. Try to import only
what you need to use.</p>
</li>
<li>
<p>To import multiple things from a module, use curly braces:
<code>use problem1::{sum, dedup};</code></p>
</li>
<li>
<p>To import all items from a module, write <code>use problem1::*</code>.</p>
</li>
<li>
<p>To import an entire module, write <code>use problem1;</code>. This form requires you to
qualify members of the module with the module name (e.g. <code>problem1::sum()</code>).
This is more verbose but does not pollute the namespace of your scope.</p>
</li>
</ol>
<p>We provided a <code>tests_provided.rs</code> file with a few test cases to start you off
with. You should add this to your library as a separate module, as you did with
each <code>problemX</code> module (but the tests don't need to be <code>pub</code>).</p>
<p>You can read more about modules in the Rust book
<a href="https://doc.rust-lang.org/book/crates-and-modules.html" rel="nofollow">here</a>.</p>

<h4><a id="user-content-testing" class="anchor" aria-hidden="true" href="#testing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Testing</h4>
<p>Before you move on, take a look at <code>tests_provided.rs</code>.</p>
<p>At the top of this file, there's a <code>#![cfg(test)]</code> attribute. <code>cfg</code>, short for
"configuration", is part of how Rust handles conditional compilation. This is
similar to (but way better than) the use of <code>#ifdef</code>s and include guards in C.
In this case, <code>#![cfg(test)]</code> tells the compiler that this module is not to be
compiled unless the <code>--test</code> flag is used with <code>rustc</code> (<code>cargo test</code> adds this
flag under the hood). This is handy because it means that you don't have to
waste time recompiling your tests every time you build unless you're actually
going to run them.</p>
<p>All of the functions in <code>test.rs</code> are annotated with the <code>#[test]</code> attribute;
this tells the compiler that they're tests. Test functions must have the
signature <code>fn() -&gt; ()</code>, or else they will not compile. Tests will be run when
you invoke <code>cargo test</code>.</p>
<p>Any test which doesn't cause a <code>panic!</code> is considered to pass. You should use
<a href="https://doc.rust-lang.org/std/macro.assert!.html" rel="nofollow"><code>assert!</code></a> or <a href="https://doc.rust-lang.org/std/macro.assert_eq!.html" rel="nofollow"><code>assert_eq!</code></a> to check guarantees and
equality (respectively).</p>
<p>As an aside, the <code>cfg</code> attribute has many other uses, like knowing
which OS or architecture you're compiling for.</p>
<p>We have provided a few tests to start you off with. You should add at least one
non-trivial test for each function that you implement. Write these in a new
